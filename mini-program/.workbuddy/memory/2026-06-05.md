## 云函数部署与超时问题修复

### 部署报错
- 问题：部署 `getCompSched` 云函数时报错 "请在编辑器云函数根目录（cloudfunctionRoot）选择一个云环境"
- 原因：`config.json` 缺少 `cloudbase.environments` 字段 + 开发者工具未绑定云环境
- 修复：① config.json 加 cloudbase 配置 ② 在 IDE 中右键 cloudfunctions/ 选择当前环境

### 超时问题
- 症状：`errCode: -504003, timed out after 3 seconds` — 云函数默认超时仅3秒，调用外部 API（football-data.org）不够用
- 修复三处：
  - `config.json`：加 `"timeout": 20` （云函数执行上限）
  - `index.js` 的 `requestApi()`：加 AbortController 15s fetch 超时（防外部API卡死）
  - `footballCloudService.js` 的 `callCloudFunction()`：加 `timeout: 20000`（客户端等待上限）
- **重要经验：修改云函数代码后必须右键→「上传并部署」才能生效，本地改动不自动同步到云端**

### 架构大重构：内存即DB + 静态数据降级

**问题根因**：免费版云函数硬限制3秒超时无法修改，外部API请求必然超时。

**解决方案 — 三层降级架构：**

1. **云函数改为读写分离**：
   - `getSchedule/todayMatches/liveScore/standings/scorers` → 全部纯读内存（<5ms，绝不超时）
   - `refreshCache/forceRefresh` → 负责调API写内存（仅定时器和手动触发调用）

2. **客户端 fallback 机制** (`footballCloudService.js`)：
   - 云函数超时/冷启动/空数据 → 自动降级到 `footballMockService`
   
3. **Mock 升级为准实时静态数据** (`footballMockService.js`)：
   - 新增 `data/scheduleStaticData.js`：从 football-data API 真实拉取的 95 场比赛、31 个比赛日、3 组积分榜
   - 新增 `scripts/fetch-schedule-static.js`：数据更新脚本，分批拉取（API限制10天/次），可随时重新执行
   - 降级时用户看到的是**真实赛程**而非假数据

4. **云函数冷启动内嵌静态数据（关键修复）**：
   - 把 staticData.js 复制到云函数目录 `cloudfunctions/getCompSched/staticData.js`
   - index.js 启动时 `require('./staticData')` 直接初始化 `_db.matches = 95场`
   - 解决多实例内存不共享问题：每个实例冷启动都有完整数据，不依赖其他实例的定时器
   - 定时器在后台无超时限制运行，成功后可热更新内存（锦上添花）

**最终验证结果**：部署后所有请求返回 `from memory-db (filtered)`，不再走 mock fallback ✅

## Bug修复：profile页面关注球队/收藏球员列表为空

**症状**：在 team-detail 点击"关注球队"后，切换到个人中心，关注球队列表仍显示空状态（"还没有关注球队"）。

**根因**：`teams2026.js` 和 `players2026.js` 的 `module.exports` 导出格式是 **对象** `{ teamsData2026: [...] }`，而非数组本身。但 `profile.js` 中 require 后直接对导出对象调 `.find()` 方法——对象没有 `.find()`，导致匹配逻辑静默失败，返回空数组。

```javascript
// ❌ 修复前（错误）
const teamsData = require('../../data/teams2026')  // → { teamsData2026: [...] } 对象
teamsData.find(...)  // undefined！对象没有 find 方法

// ✅ 修复后
const teamsAll = require('../../data/teams2026')
const teamsData = teamsAll.teamsData2026  // → [...] 数组
teamsData.find(...)  // 正常工作
```

**对比**：`team-detail.js` 正确使用了 `teamsData.teamsData2026` 访问数组属性，所以写入端没问题；只有 `profile.js` 的读取端写错了。

**修复文件**：`pages/profile/profile.js` 第1-5行 — 修正 require 引用，先取对象再取数组属性
